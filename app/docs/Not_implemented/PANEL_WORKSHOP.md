# PANEL_WORKSHOP.md

## Purpose

This file defines the Panel Workshop system for OptimalX v2.

The Panel Workshop is where users design and build custom HTML/JS panels with Eidos.
Panels are self-contained interactive tools that render inside the OptimalX editor via WebView.

Coding agents should use this file alongside AGENTIC_LOOP.md, DATA_MODEL.md, and TOOL_FUNCTIONS.md.

---

## System Structure

The Panel Workshop is a system-locked parent folder on the parent folder page.
It does not appear in the regular user folder list — accessible from the Eidos section of the app.

Inside the Panel Workshop parent folder, users create subfolders.
These are called **sub-panels** to avoid naming conflict with standard app subfolders.
Each sub-panel is one custom panel project.

---

## Sub-Panel Layout

When a user opens a sub-panel they see:

### Top header
Two dropdown menus side by side:
- **Docs** — lists all MD files for this project (README, STRUCTURE, FEATURES, FLOW, DESIGN, and any extras)
- **Code** — lists all code files for this project (HTML, JS files)

Selecting any file from either dropdown opens it in the editor area below the header.

### Editor area
Displays the currently selected file.
On first open of a new project this shows README.md with the onboarding prompts pre-filled.
When a render is ready this area switches to WebView preview.

### Chat panel
Eidos chat is accessible via the standard Eidos button — bottom sheet slides up.
User discusses, reviews, and gives feedback to Eidos here throughout the entire build.

### Start button
Visible once the user has at least filled in a description in README.md.
Tapping Start triggers Eidos to begin the build process.
Start is disabled until a description exists.

---

## Standard MD Files

Every sub-panel project gets these five files generated by Eidos before any code is written:

| File | Purpose |
|---|---|
| README.md | What this panel is, what it does, why it exists |
| STRUCTURE.md | File list and what each file does |
| FEATURES.md | Every feature, one section per feature |
| FLOW.md | User interaction flow — what happens when what is tapped |
| DESIGN.md | Layout, colors, visual rules |

Eidos adds API.md or DATA.md only if the panel requires them.

## Onboarding — README First

README.md opens automatically in a new sub-panel.
It contains three pre-filled prompts in the editor:

```
What do you want to build?

Why do you want it?

How should it work?
```

The user fills these in directly in the editor — or discusses with Eidos in chat to figure it out.
Once a description exists the Start button activates.
Eidos uses all three answers to generate the remaining MD files.
For anything it cannot determine, Eidos asks in chat while the relevant doc is open above.

## Token Notifications

The loop tracks cumulative token usage across every iteration using the token count returned by the provider on each API response.

### Milestone notifications
A notification fires at each milestone and pauses the loop with a prompt:

> "500,000 tokens used (~$0.25 with Grok). Continue or stop?"

- **Continue** — loop resumes immediately
- **Stop** — loop pauses, state saved, user returns when ready

Default milestone interval: every 200,000 tokens.
User can adjust the interval up or down in app settings.

### Approximate cost display
Each notification shows the token count and an approximate cost based on the active provider rate.

| Provider | Rate used for display |
|---|---|
| Grok 4.1 | $0.50 per 1M tokens |
| GPT-5.4 nano | $0.20 per 1M input / $1.25 per 1M output |
| Claude Haiku 4.5 | $0.50 per 1M input / $2.50 per 1M output |

Cost is approximate — displayed as a guide, not a bill.

### Hard cap
User can set a hard token cap in settings. Loop stops automatically at this number without prompting.
Default: off. User opts in.

### Resume
If the loop is paused at any milestone or cap, the reasoning note preserves full state.
Resume picks up exactly where it stopped — nothing is lost.

## Phased Build Order

### Phase 1 — Scaffold
All files created. Function shells and section comments written. No working code yet.
No render at this phase.

### Phase 2 — Layout and Design
Visual layer only — HTML structure, CSS, colors from DESIGN.md. No logic.
**First render.** User reviews visuals, gives feedback, Eidos fixes and rerenders.
Phase complete when user approves the visual.

### Phase 3 — Core Logic
Main mechanics written — primary behavior that makes the panel function.
**Second render.** User tests core behavior, gives feedback, Eidos fixes.
Phase complete when core behavior works.

### Phase 4 — Features
Each feature from FEATURES.md added one at a time.
Render → test → approve → next feature.
No feature added until the previous one is confirmed working.

### Done
Panel saved to the sub-panel. Accessible as a swipeable panel in the standard editor view.

## WebView Error Capture

Android WebView provides runtime errors via `onConsoleMessage` — error message and line number.
Errors collect silently in a buffer. Not shown to the user unprompted.

When user reports something broken:
- Error buffer plus current code file passed to Eidos
- Eidos diagnoses using error message and line number
- Eidos fixes without user needing technical detail

Logic errors (code runs but behavior is wrong) are handled through the doc loop:
user describes expected vs actual, Eidos checks FEATURES.md and FLOW.md, corrects spec then code.

---

## Edit Loop

Editing an existing panel uses the same agentic loop as a new build.
Eidos reads current files and the reasoning note, user describes the change, loop enters at the correct phase.
Token estimate provided before changes begin.

---

## Agentic Loop

The Panel Workshop runs on the AgentByte loop defined in AGENTIC_LOOP.md.

- Doc phase: Rook leads — logical, structured, one correct path through the docs
- Code phase: Queen leads — decisive, confident execution against a clear spec
- Errors: Knight activates automatically
- File overwrites and token cap: King triggers

## What Does Not Exist in v1

- No syntax highlighting or inline error indicators
- No terminal
- No external API calls from panels
- No asset or image import for panels
- No panel sharing or export
- No widget access to the Workshop
